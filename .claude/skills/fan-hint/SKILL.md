# fan-hint

## Overview

在用户调用 `/fan-hint` 时，读取当前里程碑文件，定位卡住的具体位置，自适应决定提示深度和是否给出答案。

## 第一步：判断是否有选中区域

用户调用 `/fan-hint` 时，**检查调用参数里是否包含选中的代码片段**（通常出现在命令后面，或以引用/缩进块的形式附带）。

- **有选中区域** → 直接针对选中区域给提示，跳过全文扫描
- **没有选中区域** → 执行下面的全文扫描流程

## 第二步：读代码（无选中区域时）

用 Read 读当前里程碑文件，**扫描所有 TODO 位置**，区分：
- 已经写了实质内容的（不是 `pass` 或空白）
- 还是 `pass` / 未动的

**只针对用户已经动笔但卡住的位置给提示**，完全没写的函数不要主动提示——用户可能故意留着从后往前写，或者还没到那里。

## 第二步：判断情况，决定提示策略

```
用户完全没写 → 只给概念方向，不给代码骨架
用户写了一部分但卡住 → 给出骨架，留关键表达式为空
用户多次求助同一处 / 写的已经很接近 → 可以直接给出完整写法并解释
```

**核心原则：** 提示深度随用户进展自适应，不是固定递进，最终目标是用户理解，不是永远不给答案。

## 提示格式

**无论哪种情况，都必须给出最小的真实代码片段**，不能只用文字描述。区别在于留空多少：

**用户刚开始 / 方向错误：**
```python
# 只展示关键的一两行，其余用 ... 省略
proc.stdin.write(...)  # ← 写什么？
proc.stdin.flush()
```

**用户写了一部分但卡住：**
```python
# 紧贴用户已有代码，只补关键缺失行
proc.stdin.write(json.dumps(request_body) + "\n")
proc.stdin.flush()
response = ...  # ← 从哪里读？
```

**用户多次求助同一处 / 已经很接近：**
直接给出完整实现 + 一句话解释为什么这样写。

## 禁止

- 一次性指出所有问题，只聚焦当前卡住的那一处
- 提到"对标 源码 第 N 行"

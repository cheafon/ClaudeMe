## 检查流程

### 第一步：读取进度 + 里程碑代码

**必须用 Read 工具重新读取以下两个文件，不得使用上下文中已有的版本——用户可能在上一次工具调用之后修改了文件。**

1. 从当前项目的 memory 目录读取 `project_pyagent_progress.md`，确认当前里程碑编号（路径通过 MEMORY.md 索引找到，不要硬编码绝对路径）
2. 读取对应的里程碑文件（如 `py-agent/milestones/m2_repl.py`）

### 第二步：找 TS 源码对照

根据当前里程碑的核心概念，去 `claude-code-sourcemap/restored-src/src/` 里找对应实现，作为评分基准。

根据里程碑要点自行定位对应文件和函数，读取关键源码片段，理解其设计思想（数据结构、流程顺序、边界处理），再用这些思想来评判用户代码。

### 第三步：评分

**不使用固定维度**，而是根据 rust 源码揭示的核心思想，判断用户代码：
自行拆分若干评分项，总分 100，每项说明扣分原因时引用 rust 源码的对应逻辑作为依据。

### 第四步：输出格式

```
## 里程碑进度

███████░░░ 70%  M2: 多轮 REPL + System Prompt

### 存在的问题

1. **[问题描述]**（第 N 行）— [指出与 rust 实现的差异，给思考方向，不直接给答案]
2. ...

### 做得对的地方

- ...
```

### 第五步：得分 100% 时 — 冒烟测试 + 设计思想验证

**先跑冒烟测试：**

评分达到 100% 后，**不要立即宣布完成**，先用 `printf ... | python` 管道模拟一次真实交互：

```bash
printf "<模拟输入>\nexit\n" | (cd py-agent && uv run milestones/mx_xxx.py) 2>&1
```

**CWD 必须是 `py-agent/`**，不能从项目根目录运行——里程碑脚本的配置加载（`.fan/settings.json`）依赖 CWD。

根据里程碑核心功能设计测试输入，**冒烟测试必须让核心新功能实际被触发**，而不只是验证程序能启动：
- **M3（工具调用）** → 让模型读一个真实存在的文件，检查 tool_use 是否触发、文件内容是否出现在回复里
- **M4（写文件/grep）** → 让模型写一行内容到临时文件，再读回来验证
- **M8（上下文压缩）** → 阈值 6000 tokens 靠一句话无法触发。应写一个独立测试脚本，临时调低阈值（如 10），发多轮消息强制触发压缩，再问一个依赖压缩前信息的问题，验证模型仍能回答
- **其他有阈值/条件触发的功能** → 同理：临时调低阈值或构造足够触发条件的输入，确保功能真正运行过

测试通过（输出符合预期）→ 进入设计思想验证；  
测试失败（报错或功能没触发）→ 降分并说明，不更新 memory。

**冒烟测试通过后，重新读 Rust 源码确认核心思想：**

重新读取该里程碑对应的 Rust 源文件（不是依赖记忆），对照检查用户代码是否抓住了**核心设计思路**，标准是"方向对、能跑通"，不追求完美实现：
- 核心数据流是否正确（如消息以正确顺序组装、结果正确回传）
- 主要流程是否到位（如 agentic loop 的循环结构、压缩的检测-摘要-注入流程）

**不需要关注细节缺陷**（如边界 corner case、性能优化、错误处理完备性）——这些留到所有里程碑完成后的进阶阶段处理。

核心思路正确 → 直接继续宣布完成，不需要停下来确认。

如果得分达到 100%，额外输出：

```
## 项目总进度

██░░░░░░░░ 17%  （2/17 里程碑完成）
```

然后列出所有已完成的里程碑（从 memory 读取），每个单独一行：

```
M1 流式对话 ✅
M2 多轮 REPL ✅
M3 工具调用 ✅
...
```

再用一句话衔接到下一个里程碑，格式为：
🏅 **[当前里程碑一句话总结它解决了什么] → 但现在还有 [痛点/局限] → M? 用 [核心手段] 解决这个问题。**

例如：M2 让 fan 能多轮对话了，但它只能聊天，没有能力主动读写文件 → M3 引入工具调用，让 fan 第一次能真正「动手」。

并更新 memory 文件中该里程碑的状态为 ✅ 完成，填写完成日期。

最后用 AskUserQuestion 工具询问（需先用 ToolSearch 加载）：

```
question: "准备好开始 M? 了吗？还是我们先讨论一下？"
header: "下一步"
options:
  - label: "开始 M?"      description: "生成下一个里程碑脚手架，开始实现"
  - label: "自己跑一下"   description: "cd py-agent && uv run milestones/mX_xxx.py — 先自己体验一下刚完成的功能"
  - label: "先休息"       description: "今天先到这，下次再继续"
```

用户选择"先休息"或"自己跑一下"时，额外输出运行命令，方便用户随时自己跑：

```
`cd py-agent && uv run milestones/mX_xxx.py`
```

项目总进度 = 已完成里程碑数 / 26。

### 进度条格式

- 10 格，每格代表 10%
- 已完成用 `█`，未完成用 `░`
- 四舍五入到最近的 10%
- 格式：`██████░░░░ 60%`

### 反馈原则

- 读 Rust 源码是内部评判依据，但反馈给用户时用自己的话描述问题，不出现"对标 Rust xxx 行"之类的引用
- 指出错误的位置和类型，不直接给出修正后的代码
- 鼓励性语气
